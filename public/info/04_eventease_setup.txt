WHAT WAS BUILT
==============
EventEase is a complete event reservation system. Visitors browse upcoming
events, book tickets, cancel and rebook reservations. Organizers manage their
own events (create, edit, delete). An admin manages all events and assigns
user roles.

All EventEase code is fully isolated from the two other practice projects
(TechBazaar / DevTalk) — separate database tables, a dedicated role column,
its own middleware, and its own route file.

DATABASE DESIGN DECISIONS
==========================

1. WHY decimal(8,2) FOR PRICE — NOT float
   ----------------------------------------
   Floating-point types (MySQL FLOAT / DOUBLE) represent numbers in binary.
   They cannot store some decimal fractions exactly:
     0.1 + 0.2  →  0.30000000000000004  (binary float problem)
   decimal(8,2) is a fixed-point type: it stores the number exactly as
   written. €49.99 stays €49.99 — no rounding errors.
   This is critical for money. Always use decimal for prices.

2. WHY ee_ TABLE PREFIX
   ----------------------
   Three separate projects (techbazaar, devtalk, eventease) share one
   database. Using prefixes prevents name collisions:
     ee_events, ee_reservations (EventEase)
     devtalk_threads, devtalk_posts, devtalk_categories, ... (DevTalk)
   Without prefixes a generic "events" table could silently conflict with
   future migrations in other projects.

3. WHY A SEPARATE ee_role COLUMN
   --------------------------------
   The 'role' column belongs to TechBazaar (admin / seller / buyer).
   DevTalk uses 'devtalk_role' (admin / moderator / user).
   EventEase uses 'ee_role' (admin / organizer / visitor).
   One column per project keeps all three systems completely independent:
     → A TechBazaar admin is NOT automatically an EventEase admin.
     → A DevTalk moderator is NOT automatically an EventEase organizer.
   This is clean domain isolation — important for exam casus work.

4. WHY unique(['user_id', 'event_id']) ON ee_reservations
   --------------------------------------------------------
   A user should only have ONE reservation row per event. Without this
   constraint, two rapid browser submits could insert two rows before
   either commits, resulting in double-billing.
   The composite unique index guarantees this at the database level —
   server-side checks alone are not enough due to race conditions.

5. WHY CANCEL = UPDATE (NOT DELETE)
   -----------------------------------
   When a user cancels, we UPDATE the status to 'cancelled' and keep the
   row. Benefits:
     a) Audit trail: you can see that a booking existed and was cancelled.
     b) Re-booking is simpler: UPDATE the existing row back to 'confirmed'
        instead of DELETE + re-INSERT. This avoids hitting the unique
        constraint again (INSERT would need the unique index to pass).
     c) Created_at timestamp is preserved, which is useful for analytics.

6. WHY .whereNumber('event') ON {event} ROUTES
   -----------------------------------------------
   Without this constraint, a URL like:
     GET /project/eventease/organizer/events/create
   would try to route-model-bind the string "create" as an Event ID,
   causing a 404 or ModelNotFoundException before the real route can match.
   .whereNumber('event') restricts the {event} parameter to digits only.
   The explicit /events/create route above it then catches "create" correctly.
   This is the same fix applied to DevTalk's {thread} parameter.


ARCHITECTURE OVERVIEW
=====================

  Pattern: each project is fully self-contained
  ──────────────────────────────────────────────
  ┌──────────────────────────────────────────────────┐
  │ routes/eventease.php          Route definitions  │
  │   prefix: project/eventease   + name: eventease. │
  │                                                  │
  │ Controllers/EventEase/        Business logic      │
  │   EventController             → public events    │
  │   ReservationController       → booking          │
  │   OrganizerEventController    → CRUD for org     │
  │   Admin/UserController        → role management  │
  │                                                  │
  │ Models/EventEase/             Data layer          │
  │   Event      (ee_events)                         │
  │   Reservation (ee_reservations)                  │
  │                                                  │
  │ Middleware/EnsureEventEaseRole  → 'ee_role' alias│
  │                                                  │
  │ Views: projects/eventease/    8 Blade views       │
  └──────────────────────────────────────────────────┘

  Overbooking prevention (Event model):
  ──────────────────────────────────────
  remainingCapacity() = capacity - SUM(confirmed reservations seats)
  Called in ReservationController@store before creating/restoring a booking.
  Also called in ReservationController@update (rebook path) to re-verify.

  Scope pattern (Event model):
  ─────────────────────────────
  scopeUpcoming() → WHERE date > NOW() ORDER BY date ASC
  Used by EventController@index for the public events list.
  Organizers/admins use orderByDesc('date') to see all events incl. past.

  Role middleware chain:
  ──────────────────────
  Route::middleware('auth')                    → must be logged in
  Route::middleware('ee_role:organizer,admin') → must be organizer OR admin
  Route::middleware('ee_role:admin')           → admin only
